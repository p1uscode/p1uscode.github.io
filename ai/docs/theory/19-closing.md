下の層ほど「そもそも何が起きているか」、上の層ほど「その組み合わせで何を作るか」。

## この知識で何ができるようになるか

- **新しいプロダクトを見たとき**に中身を分解できる。「これは LLM + RAG + tool calling を組み合わせたやつか」と即座に分類できる
- **問題が起きたとき**に [第 16 章](16-engineering-layers.md) の 3 層 (プロンプト / コンテキスト / ハーネス) のどこを直すか判断できる
- **自分でエージェントを作るとき**に何を考慮すべきか分かる。道具選び、停止条件、ガード、観測、評価の落としどころ
- **チームで議論するとき**に共通言語が持てる。「これは state の問題」「これは context window の問題」と名前で呼べる

## これから先の進み方

座学はこれで一区切り。あとはひたすら**手を動かして失敗しながら学ぶ**フェーズに入る。

- 本リポジトリの `examples/agent-demo/` を触り、ツールを増やしたり system prompt を書き換えたりする
- Langfuse のトレースを眺め、tool_calls の往復や token 消費を観察する ([第 07 章](07-observability.md) / [ハンズオン 2](../hands-on/langfuse-traces.md))
- 気になった章に戻って読み直す。各章は独立して読めるように書いてある
- 自分のユースケース (仕事 / 趣味 / 研究) で試し、うまく行かなかったら対応する章に戻る

座学資料は**辞書 / 索引として**使うのが本来の使い方。頭から通読するより、困ったときに該当章を開く方が定着しやすい。

## 最後に

LLM / エージェントの世界はプロダクトの顔ぶれが毎月変わる。しかし**核となる仕組みは数年単位で安定している** ([第 18 章](18-ai-tools-overview.md) の終盤で触れた通り)。新しいツールが出てきても、章 00-17 の原理を押さえていれば「これは既存の何の組み合わせか」と分類できる。

積み上げた知識は long-lived な投資である。焦って最新プロダクトを追うより、仕組みを理解して全体像を俯瞰できる状態を維持する方が、結果的に早い。
